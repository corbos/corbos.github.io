Lesson: HTML Forms

Introduction

Much of HTML is intended for displaying data. HTML forms and form controls allow users to enter data. Once we can enter data, we can build proper web applications. Without forms, the web would be a collection of read-only brochures. With forms, we can send data to an HTTP server where it's processed, stored, and used to calculate a result. Even without sending data, forms are valuable. Selecting from a list of choices might display a focused UI. Checking or unchecking a checkbox shows and hides details. Forms drive interaction.

Without forms, Google Search wouldn't be possible. Wikipedia, Pinterest, and Twitch wouldn't exist. Even largely read-only sites like the Wall Street Journal and Youtube (stream-only) wouldn't exist without the ability to log in with a username input, password input, and a submit button.

Learning Outcomes

When you've finished this lesson and its exercises, you should be able to:

  • Describe the problem forms solve.
  • Name 5 common form controls and describe how a user interacts with them.

Form Structure

An HTML form is a block element. It's not useful without children because the children are what allow user interaction. Children include HTML elements like label, input, select, textarea, and button. These elements are called form controls. A control is a built-in UI component that supports one of several standard user interactions: clicking a button, selecting from a dropdown, entering text, etc. Children may also include HTML layout and text elements.

<form action="/relative/url" method="post">
    <div class="mb-3">
        <label for="username" class="form-label">Username</label>
        <input type="text" id="username" name="username" class="form-control" autocomplete="off">
    </div>
    <div class="mb-3">
        <label for="password" class="form-label">Password</label>
        <input type="password" id="password" name="password" class="form-control">
    </div>
    <div class="mb-3">
        <button class="btn btn-primary" type="submit">Submit</button>
    </div>
</form>

Form Submission

Without JavaScript, a form submits its data via an HTTP request to a server. The action attribute specifies the request URL. If it's omitted, the action is the URL in the browser's address bar. The method attribute specifies the request method: GET or POST. No other methods are valid in a non-JavaScript form.

Form submission is triggered by:

  1. Clicking a button type="submit" that's inside the form or has its form attribute set.
  2. Clicking an input type="submit" that's inside the form.
  3. Pressing the access key sequence for a submit input's accesskey value. (Uncommon)
  4. Pressing Enter while focused on a form control inside a form and a submit element exists. This behavior helps power users enter data very quickly. They tab through form controls, entering data as they go, and hit Enter when they're done. There's no need to use the mouse.

Common Form Controls

Create a new HTML document with the following content.

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Forms</title>
</head>
<body>
    <!-- Add snippets here. -->
</body>
</html>

Copy and paste snippets into the body. Open the document in the browser and review each snippet.

label

(Some might argue that the label element isn't a form control. It doesn't accept data.)

A label associates text to a form control. A label is linked to a control by assigning the control's id value to the label's for. Clicking the label puts focus on its input and can even check or uncheck checkboxes.

<div>
    <label for="petName">Name</label>
    <input type="text" id="petName" name="petName">
</div>
<div>
    <label for="likesKibble">Likes Kibble</label>
    <input type="checkbox" id="likesKibble" name="likesKibble">
</div>

Copy the snippet. View in the browser. Click on the labels.

input

input is the workhorse form control. It solves different UI concerns depending on its type attribute. If the type isn't specified, the default is text. The types below are just a sample. There are many more.

type="text"

The text input allows the user to enter text on a single line.

The id attribute unique identifies a control.

The name attribute is the key used to send a control's data via an HTTP POST or GET. It's not strictly required if form submission is disabled but can still be useful.

If the value attribute is set, the control is pre-filled with its value.

If the placeholder attribute is set, the control starts with its value as a text prompt. As soon as the use enters their own text, the placeholder disappears. Placeholder text isn't part of the actual value.

Add a title to display a message when the user hovers over the control.

<input type="text" id="txtOne" name="txtOne">
<!-- default's to type="text" -->
<input id="txtTwo" name="txtTwo">
<!-- has an initial value -->
<input type="text" id="txtThree" name="txtThree" value="This is the starting value.">
<!-- placeholder prompts the user -->
<input type="text" id="txtFour" name="txtFour" placeholder="[Enter some text here.]">
<!-- title shows a message when the user hovers -->
<input type="text" id="txtFive" name="txtFive" title="I am for entering text.">

type="password"

The password type accepts text input, but the input is masked so that casual observers can't read it. The text is not protected in any way from JavaScript or form submission. The text is still plain text.

<label for="password">Password</label>
<input type="password" id="password" name="password">
<label for="passwordConfirm">Confirm Password</label>
<input type="password" id="passwordConfirm" name="passwordConfirm">

type="checkbox"

The checkbox input allows the user to check or uncheck a single checkbox.

If the value attribute is set, it's the value sent via HTTP POST or GET if the checkbox is checked. If the checkbox isn't checked, no value is sent. The default value is "on".

Use the stand-alone checked attribute to pre-check a checkbox.

Checkboxes have no text content. It's a single, lonely checkbox. Be sure the user knows what a checkbox is for with a label or other text indicator. The title attribute probably isn't sufficient.

<input type="checkbox" id="chkOne" name="chkOne">
<input type="checkbox" id="chkTwo" name="chkTwo" value="isHopeful">
<input type="checkbox" id="chkThree" name="chkThree" checked>
<label for="chkFour">I Am Check Four</label>
<input type="checkbox" id="chkFour" name="chkFour" title="Check me to win a prize." checked>

type="radio"

The radio input allows the user to press (or select) a single button out of a group. Old car radios had physical buttons for presetting FM/AM frequencies. Only one button could be pressed down at a time. New radios operate the same way -- there's no way to tune the radio to two frequencies -- but we don't get the physical feedback of a mashed down radio button.

The name attribute links a set of radio buttons so that only one with the same name can be selected at a time. When another is selected, any selected radio button with the same name is unselected.

If the value attribute is set, it's the value sent via HTTP POST or GET when the radio is selected. If the radio isn't selected, no value is sent.

Use the checked attribute to pre-select a radio button.

Radio buttons have no text content so it's important to let a user know how to use them.

<div>
    <h2>Choose a primary color.</h2>
    <input type="radio" name="color" id="rdRed" value="Red">
    <input type="radio" name="color" id="rdBlue" value="Blue">
    <input type="radio" name="color" id="rdYellow" value="Yellow">
</div>
<div>
    <h2>How do you commute?</h2>
    <label for="rdWalk">Walk</label>
    <input type="radio" name="commute" id="rdWalk" value="Walk">
    <label for="rdBike">Bike</label>
    <input type="radio" name="commute" id="rdBike" value="Bike">
    <label for="rdCar">Car</label>
    <input type="radio" name="commute" id="rdCar" value="Car">
    <label for="rdBus">Bus</label>
    <input type="radio" name="commute" id="rdBus" value="Bus">
    <label for="rdTrain">Train</label>
    <input type="radio" name="commute" id="rdTrain" value="Train">
    <label for="rdHome">Work from home</label>
    <input type="radio" name="commute" id="rdHome" value="Work From Home" checked>
</div>

type="number"

The number input restricts text content to a number. Its value attribute is still a string. When we read its value via JavaScript: document.getElementById("id").value, we still get a string and must convert to number if desired.

The min attribute sets the lowest number allowed.

The max attribute sets the highest number allowed.

The number input includes up and down arrow buttons to increase or decrease the number without typing. The step attribute set the arrow increment value. The default is 1.

min, max, and step don't prevent the user from typing other numbers. Validation is still required.

<input type="number" id="anyNumber" name="anyNumber">
<input type="number" id="between1and10" name="between1and10" value="1" min="1" max="10" step="0.1">

select

The select element allows a user to select one or more options from a fixed set. In this way, it's not unlike a radio button. Its layout is more compact and, unlike radio buttons, it can be configured to select multiple options.

Select represents data options with <option> child elements. Each option renders the text inside it and can optionally use the value attribute as the actual value selected. When value is set, text is a prompt and value is the selected value.

Use the selected attribute to pre-select an option.

<!-- no values, text is the value -->
<select name="color" id="color">
    <option>Red</option>
    <option selected>Blue</option>
    <option>Yellow</option>
</select>

<!-- explicit values -->
<select name="commute" id="commute">
    <option value="walk">Walk</option>
    <option value="bike">Bicycle</option>
    <option value="car">Drive</option>
    <option value="bus">Bus</option>
    <option value="train">Train</option>
    <option value="home" selected>Work from Home</option>
</select>

Select can also be configured to select multiple items from a list by adding the multiple attribute.

<h2>Which commuting methods would you consider?</h2>
<select name="commute" id="commute" multiple>
    <option value="walk">Walk</option>
    <option value="bike">Bicycle</option>
    <option value="car">Drive</option>
    <option value="bus">Bus</option>
    <option value="train">Train</option>
    <option value="home">Work from Home</option>
</select>

textarea

The textarea element allows users to enter multi-line text in a multi-line input format.

Unlike other controls, textarea doesn't preset its content with the value attribute. It embeds the text inside the tag.

The cols attributes sets the width of a textarea in average characters. The rows attribute sets the height in lines of text. The element can also be styled with CSS.

<textarea id="comment" name="comment"></textarea>
<textarea id="profile" name="profile" cols="35" rows="10">My favorite color is light tan.
I like puppies.</textarea>

button

The button element has the simplest user interaction. It allows a user to click it in the hope that something will happen.

A button is a container element. It can include any valid HTML inside of it. It has three types.

type="submit"

The submit button submits a form. It must be included inside the form or set the form attribute to trigger form submission.

<!-- submit button inside the form -->
<form onsubmit="alert('first form submitted');return false;">
    <button type="submit">Submit</button>
</form>

<!-- submit button with the `form` attribute -->
<form id="frmTwo" onsubmit="alert('second form submitted');return false;">
</form>
<button type="submit" form="frmTwo">Submit</button>

type="button"

The button type triggers an action, but it's up to JavaScript to handle it. On its own, the button type doesn't do anything.

type="reset"

A reset button resets a form's controls. It blanks them out. Understandably, most people hate this. When they click the reset button, it's usually a mistake. Best to avoid the reset button unless there's a really good reason to use it.

Conclusion

Forms and form controls are what power true web applications. Forms let people create things (and check their account balance).
